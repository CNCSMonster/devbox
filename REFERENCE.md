# Devbox Reference

Advanced documentation for devbox internals. See [SKILL.md](SKILL.md) for the agent-facing quick start.

## Naming Strategy

Default project name = `slug(absolutePath) + "-" + hash10`.

Example: `/home/alice/work/client-a/backend` → `home-alice-work-client-a-backend-a1b2c3d4e5`.

### Slug rules

1. Strip leading `/`.
2. Lowercase.
3. Replace non `a-z0-9` with `-`.
4. Collapse consecutive `-`.
5. Strip leading/trailing `-`.
6. Empty result → `project`.

### Hash rules

```bash
hash_input="${host}:${uid}:${docker_context}:${absolute_path}"
hash="$(printf '%s' "$hash_input" | sha256sum | cut -c1-10)"
```

- `absolute_path` = `realpath "$PWD"`.
- Different machines, users, or Docker contexts → different hash → no collision.

## Docker Resource Names

From project name `${name}`:

```text
composeProject = ${name}
container      = ${name}-dev
imageName      = ${name}:dev       # reserved config field; image mode uses config.image directly
volumeName     = ${name}-dev-home
```

Volume names use hyphens, not underscores.

## Docker Ownership Labels

Both container and volume carry `devbox.*` labels:

```yaml
labels:
  devbox.managed: "true"
  devbox.version: "1"
  devbox.name: "${name}"
  devbox.owner_uid: "${uid}"
  devbox.identity_hash: "sha256:..."
```

Labels are internal metadata. Devbox checks labels before operating on existing Docker resources.

## Conflict Detection

| Existing resource | Labels | Action |
|-------------------|--------|--------|
| Does not exist | — | Allow |
| Exists, labels match | ✅ | Reuse |
| Exists, labels mismatch | ❌ | Refuse |
| Exists, no labels | ❌ | Refuse |

No command, including `init --force`, may take ownership of unlabeled or mismatched resources.

## Config Schema

`.devbox/config.json` v1 identity model:

```json
{
  "version": 1,
  "name": "home-alice-work-client-a-backend-a1b2c3d4e5",
  "displayName": "/home/alice/work/client-a/backend",
  "container": "home-alice-work-client-a-backend-a1b2c3d4e5-dev",
  "composeProject": "home-alice-work-client-a-backend-a1b2c3d4e5",
  "imageName": "home-alice-work-client-a-backend-a1b2c3d4e5:dev",
  "volumeName": "home-alice-work-client-a-backend-a1b2c3d4e5-dev-home",
  "image": "ghcr.io/cncsmonster/dotfiles:latest",
  "type": "python",
  "shell": "zsh",
  "identity": {
    "strategy": "full-path-hash-v1",
    "projectRoot": "/home/alice/work/client-a/backend",
    "identityHash": "sha256:a1b2c3d4e5f6...64 hex chars",
    "ownerUid": "1000",
    "host": "hostname",
    "dockerContext": "default"
  },
  "createdAt": "2026-05-26T23:00:00+08:00"
}
```

## docker-compose.yml Template

Default image mode generated by `devbox init`:

```yaml
name: ${name}

services:
  dev:
    image: ${image}
    container_name: ${container}
    working_dir: /app
    labels:
      devbox.managed: "true"
      devbox.version: "1"
      devbox.name: "${name}"
      devbox.owner_uid: "${uid}"
      devbox.identity_hash: "${hash}"
    env_file:
      - path: ../.env
        required: false
    extra_hosts:
      - "host.docker.internal:host-gateway"
    environment:
      PATH: /root/.local/share/mise/installs/node/latest/bin:/root/.local/share/mise/shims:/root/.local/bin:/root/.cargo/bin:/opt/project-venv/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/snap/bin
    volumes:
      - ..:/app
      - dev-home:/root
      - /app/.devbox
    stdin_open: true
    tty: true
    command: sleep infinity

volumes:
  dev-home:
    name: ${volume_name}
    labels:
      devbox.managed: "true"
      devbox.version: "1"
      devbox.name: "${name}"
      devbox.owner_uid: "${uid}"
      devbox.identity_hash: "${hash}"
```

Key properties:

- top-level `name:` equals the project name.
- container and volume both have `devbox.*` ownership labels.
- `.devbox/` is hidden inside the container with an anonymous mount at `/app/.devbox`.
- host HOME or user config directories are not mounted.
- no project-local Dockerfile is required in default image mode.

## Optional Build Mode

If a project explicitly needs build-time customization, humans or agents may add a devbox-specific Dockerfile at:

```text
.devbox/image/Dockerfile
```

Use it for system packages, language runtimes, global tools, or cacheable dependencies that do not contain host secrets. Runtime/host-specific setup belongs in `.devbox/runtime/container-init.sh`.

## Host-side Container Init Script

Generated at `.devbox/runtime/container-init.sh`. This script runs on the host after the container has been created and started.

```bash
.devbox/runtime/container-init.sh <container-name>
```

It may use `docker cp` and `docker exec` to initialize the container. It should create `/tmp/.container-init-executed` inside the container only after successful bootstrap.

## Command Semantics

- `devbox enter`: create/start container, run host-side bootstrap only if marker is missing, then attach shell.
- `devbox restart`: restart/create container, run bootstrap only if marker is missing, then attach shell.
- `devbox rebuild`: remove and recreate container, start it, run bootstrap, preserve HOME volume, do not attach shell.
- `devbox clean`: remove container and HOME volume after confirmation.

## Host Service Access

Inside container, use `host.docker.internal` for host services:

```bash
curl http://host.docker.internal:8989/v1/models
```

Host services must listen on `0.0.0.0` or Docker gateway, not just `127.0.0.1`.

## Old Project Compatibility

v1 does not support old-format configs or unlabeled Docker resources. If `.devbox/config.json` lacks the v1 identity model, devbox refuses and prompts `devbox init --force`.

`--force` only overwrites local `.devbox/` files. It never bypasses Docker ownership checks.

## Verification Checklist

1. `devbox init` succeeded.
2. `devbox config` shows valid v1 config.
3. Default compose uses `image: ${image}` and does not require `.devbox/runtime/Dockerfile`.
4. Container and volume have `devbox.*` labels.
5. Host-side bootstrap marker exists in initialized containers: `/tmp/.container-init-executed`.
6. `devbox verify` passes all checks.
